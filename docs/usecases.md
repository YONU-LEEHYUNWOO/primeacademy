## 🧩 3. 상세 유스케이스 시나리오 (Detailed Use Case Scenarios)

### ✅ UC-04 — **상담 예약**

**목적:**
방문자가 간단한 폼을 통해 학원 상담을 예약할 수 있도록 한다.

**주 액터:** 방문자
**보조 액터:** 웹 시스템, 운영자

---

### 📋 사전 조건 (Preconditions)

* 사용자가 랜딩페이지에 접속함 (로그인 불필요)
* JavaScript 활성화 상태
* 이메일 전송 API (EmailJS 또는 FormSubmit 등) 정상 동작

### 📋 사후 조건 (Postconditions)

* 예약 데이터가 운영자 이메일로 전송됨
* 방문자에게 “예약 완료” 메시지 표시

---

### 📜 주요 단계 (Main Steps and Flow of Events)

| 단계 | 액터    | 시스템 | 설명                            |
| -- | ----- | --- | ----------------------------- |
| 1  | 방문자   |     | 상단 또는 하단의 “상담 예약하기” 버튼 클릭     |
| 2  | 시스템   |     | `/reservation` 페이지 로드         |
| 3  | 방문자   |     | 이름, 연락처, 상담 희망일, 문의 내용 입력     |
| 4  | 시스템   |     | 폼 입력값 유효성 검사 (빈값, 형식 체크)      |
| 5  | 방문자   |     | “제출하기” 클릭                     |
| 6  | 시스템   |     | 입력값 검증 성공 시 이메일 전송 API 호출     |
| 7  | 웹 시스템 |     | 관리자 이메일로 데이터 전송               |
| 8  | 시스템   |     | 성공 메시지 표시 (“상담 신청이 완료되었습니다.”) |
| 9  | 운영자   |     | 이메일 수신 후 수동으로 연락 및 일정 확정      |

## 🔄 4. 대체 흐름 및 엣지 케이스 (Alternative Flows and Edge Cases)

| 상황                           | 설명                      | 처리 방식                                   |
| ---------------------------- | ----------------------- | --------------------------------------- |
| **AF-01:** 필수 입력 누락          | 사용자가 이름 또는 연락처를 입력하지 않음 | “필수 항목을 입력해주세요.” 표시 후 제출 중단             |
| **AF-02:** 잘못된 연락처 형식        | 숫자가 아닌 문자 포함            | “연락처 형식이 올바르지 않습니다.” 경고 표시              |
| **AF-03:** 이메일 전송 실패         | 네트워크 오류 또는 API 오류       | “서버 오류가 발생했습니다. 잠시 후 다시 시도해주세요.” 메시지 표시 |
| **AF-04:** 모바일 환경에서 키보드 오버레이 | 폼 입력 중 하단 CTA 버튼 가려짐    | 입력 포커스 시 버튼 고정 해제                       |
| **AF-05:** 중복 제출             | 사용자가 버튼을 여러 번 클릭        | 첫 클릭 이후 버튼 비활성화 처리                      |

## ⚙️ 5. 비즈니스 규칙 및 제약 (Business Rules and Constraints)

| 규칙 ID | 규칙 설명                                           |
| ----- | ----------------------------------------------- |
| BR-01 | 모든 예약 데이터는 실시간 전송이 아닌 이메일 기반으로 처리한다.            |
| BR-02 | 데이터베이스 저장 없이, Local Form → Email API 방식으로 한정한다. |
| BR-03 | 하루 최대 예약 건수는 10건으로 제한 (운영자 대응 가능 범위 내).         |
| BR-04 | 예약 폼은 이름, 연락처, 상담일, 문의내용 4가지 항목만 허용한다.          |
| BR-05 | 예약 데이터는 서버에 저장하지 않으며, 개인정보 보호정책 고지 필요.          |

## 🚨 6. 예외 처리 절차 (Exception Handling Procedures)

| 예외 상황           | 탐지 방법               | 사용자 알림              | 복구 절차          |
| --------------- | ------------------- | ------------------- | -------------- |
| 이메일 전송 실패       | API 응답 코드 400~500   | “전송 실패, 다시 시도해주세요.” | 재전송 버튼 활성화     |

| 네트워크 연결 끊김      | Fetch 오류            | “인터넷 연결을 확인해주세요.”   | 자동 재시도 3회      |
| JavaScript 비활성화 | 스크립트 로딩 실패          | 경고 배너 표시            | 기본 전화 문의 안내 표시 |
| 스팸 데이터 입력       | 비정상적인 입력 탐지 (Regex) | “입력값을 다시 확인해주세요.”   | 제출 중단          |

## ✅ 11. 결론

본 Use Case 문서는 **“학부모 중심의 상담 예약 중심 랜딩페이지”** 설계를 기반으로
사용자 여정, 폼 제출 흐름, 오류 처리 및 보안 정책을 상세히 정의합니다.

> 📌 핵심 목표:
> 방문자가 **정보를 신뢰 → 즉시 예약 전환**하도록
> **단순하면서 신뢰성 높은 UX 흐름**을 보장하는 것.

Next.js 정적 구조와 Email API 연동을 통해
**운영 효율성과 유지보수 편의성**을 극대화하며,
데이터 저장 없이 개인정보 보호 수준을 높인 **경량형 학원 랜딩 솔루션**입니다.
